# Project Rules

## 1. Tech Stack

* Next.js (App Router)
* React 19+
* TypeScript (strict mode)
* Zustand (state management)
* CSS Modules + SCSS

---

## 2. General Principles

* Код должен быть простым, читаемым и предсказуемым
* Не писать "умный" код ради умности
* Предпочитать явность вместо магии
* Минимизировать дублирование
* Всегда думать о масштабировании

---

## 3. Project Structure

src/

* app/ → страницы (routing)
* components/

  * ui/ → переиспользуемые UI компоненты (Button, Input)
  * layout/ → layout компоненты (Header, Footer, Container)
  * composite/ → составные компоненты
* features/ → бизнес-логика (standart, training, program)
* store/ → Zustand stores
* lib/ → утилиты и бизнес-функции
* config/ → константы, справочники
* utils/ → вспомогательные функции

---

## 4. Component Rules

### Разделение:

* UI компоненты → не содержат бизнес-логики
* Feature компоненты → содержат бизнес-логику
* Layout → только структура страницы

### Требования:

* Каждый компонент = одна ответственность
* Обязательная типизация props
* Минимум логики в JSX
* Логика выносится в hooks или lib

---

## 5. State Management (Zustand)

* Использовать create() для стора
* Не мутировать state напрямую
* Всегда использовать set((state) => ...)
* Хранить только нужные данные (не дублировать)
* Бизнес-логика может находиться внутри store

### Пример:

```ts
set((state) => ({
  count: state.count + 1
}))
```

---

## 6. Business Logic

* Вынесена в:

  * store/
  * lib/
* Не хранить бизнес-логику в компонентах
* Повторяющаяся логика → обязательно выносится

---

## 7. Naming Convention

* Переменные → camelCase
* Компоненты → PascalCase
* Файлы компонентов → kebab-case или PascalCase (один стиль на проект)
* Константы → UPPER_CASE

---

## 8. TypeScript Rules

* Не использовать `any`
* Использовать строгую типизацию
* Создавать типы для:

  * props
  * store
  * API данных

---

## 9. Styling

* Только CSS Modules
* SCSS для расширенных возможностей
* Использовать helpers (fluid, mixins)
* Не использовать inline-стили без необходимости

---

## 10. Reusability

Перед созданием нового кода проверить:

* Можно ли переиспользовать существующий компонент?
* Есть ли уже похожая логика?
* Можно ли вынести в универсальную функцию?

---

## 11. File Placement Rules

Перед созданием файла определить:

* UI → components/ui
* Feature → features/*
* Store → store/
* Utils → utils/
* Config → config/

Нельзя создавать файлы вне структуры

---

## 12. Data Flow

* Данные идут сверху вниз
* Store используется как единый источник правды
* Не дублировать данные в разных местах

---

## 13. Performance

* Не создавать лишние ререндеры
* Использовать мемоизацию при необходимости
* Не оптимизировать преждевременно

---

## 14. Error Handling

* Проверять входные данные
* Обрабатывать undefined / null
* Не допускать падений приложения

---

## 15. AI Interaction Rules

Если код генерируется с помощью ИИ:

* Проверять соответствие RULES.MD
* Не принимать код “как есть”
* Упрощать, если код слишком сложный
* Проверять типизацию

---

## 16. Forbidden

Запрещено:

* Использовать any без причины
* Писать бизнес-логику в UI компонентах
* Дублировать код
* Ломать существующую архитектуру
* Добавлять библиотеки без необходимости

---

## 17. Goal

Цель проекта:

* Чистая архитектура
* Масштабируемость
* Понятный код
* Быстрая разработка
* Минимум багов
